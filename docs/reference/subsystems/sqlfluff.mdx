The Sqlfluff SQL linter (https://github.com/sqlfluff/sqlfluff).

This version of Pants uses `sqlfluff` version 2.3.5 by default. Use a dedicated lockfile and the `install_from_resolve` option to control this.

Backend: <span className="color--primary">`pants.backend.experimental.sql.lint.sqlfluff`</span>

Config section: <span className="color--primary">`[sqlfluff]`</span>

## Advanced options

### `config`

<Option
  cli_repr={`--sqlfluff-config=<file_option>`}
  env_repr='PANTS_SQLFLUFF_CONFIG'
  toml_repr={`[sqlfluff]
config = <file_option>`}
  default_repr={`None`}
>

Path to the `pyproject.toml` or `.sqlfluff` file to use for configuration (https://docs.sqlfluff.com/en/stable/configuration.html).

Setting this option will disable `[sqlfluff].config_discovery`. Use this option if the config is located in a non-standard location.

</Option>

### `config_discovery`

<Option
  cli_repr={`--[no-]sqlfluff-config-discovery`}
  env_repr='PANTS_SQLFLUFF_CONFIG_DISCOVERY'
  toml_repr={`[sqlfluff]
config_discovery = <bool>`}
  default_repr={`True`}
>

If true, Pants will include any relevant config files during runs (`pyproject.toml`, and `.sqlfluff`).

Use `[sqlfluff].config` instead if your config is in a non-standard location.

</Option>

### `console_script`

<Option
  cli_repr={`--sqlfluff-console-script=<str>`}
  env_repr='PANTS_SQLFLUFF_CONSOLE_SCRIPT'
  toml_repr={`[sqlfluff]
console_script = <str>`}
  default_repr={`sqlfluff`}
>

The console script for the tool. Using this option is generally preferable to (and mutually exclusive with) specifying an `--entry-point` since console script names have a higher expectation of staying stable across releases of the tool. Usually, you will not want to change this from the default.

</Option>

### `entry_point`

<Option
  cli_repr={`--sqlfluff-entry-point=<str>`}
  env_repr='PANTS_SQLFLUFF_ENTRY_POINT'
  toml_repr={`[sqlfluff]
entry_point = <str>`}
  default_repr={`None`}
>

The entry point for the tool. Generally you only want to use this option if the tool does not offer a `--console-script` (which this option is mutually exclusive with). Usually, you will not want to change this from the default.

</Option>

### `install_from_resolve`

<Option
  cli_repr={`--sqlfluff-install-from-resolve=<str>`}
  env_repr='PANTS_SQLFLUFF_INSTALL_FROM_RESOLVE'
  toml_repr={`[sqlfluff]
install_from_resolve = <str>`}
  default_repr={`None`}
>

If specified, install the tool using the lockfile for this named resolve.

This resolve must be defined in `[python].resolves`, as described in https://www.pantsbuild.org/2.24/docs/python/overview/lockfiles#lockfiles-for-tools.

The resolve&#x27;s entire lockfile will be installed, unless specific requirements are listed via the `requirements` option, in which case only those requirements will be installed. This is useful if you don&#x27;t want to invalidate the tool&#x27;s outputs when the resolve incurs changes to unrelated requirements.

If unspecified, and the `lockfile` option is unset, the tool will be installed using the default lockfile shipped with Pants, which uses `sqlfluff` version 2.3.5.

If unspecified, and the `lockfile` option is set, the tool will use the custom `sqlfluff` &#x22;tool lockfile&#x22; generated from the `version` and `extra_requirements` options. But note that this mechanism is deprecated.

</Option>

### `interpreter_constraints`

<Option
  cli_repr={`--sqlfluff-interpreter-constraints="['<str>', '<str>', ...]"`}
  env_repr='PANTS_SQLFLUFF_INTERPRETER_CONSTRAINTS'
  toml_repr={`[sqlfluff]
interpreter_constraints = [
    '<str>',
    '<str>',
    ...,
]`}
  default_repr={`[\n  "CPython>=3.8,<4"\n]`}
>

Python interpreter constraints for this tool.

</Option>

### `requirements`

<Option
  cli_repr={`--sqlfluff-requirements="['<str>', '<str>', ...]"`}
  env_repr='PANTS_SQLFLUFF_REQUIREMENTS'
  toml_repr={`[sqlfluff]
requirements = [
    '<str>',
    '<str>',
    ...,
]`}
  default_repr={`[]`}
>

If `install_from_resolve` is specified, install these requirements, at the versions provided by the specified resolve&#x27;s lockfile.

Values can be pip-style requirements (e.g., `tool` or `tool==1.2.3` or `tool>=1.2.3`), or addresses of `python_requirement` targets (or targets that generate or depend on `python_requirement` targets). Make sure to use the `//` prefix to refer to targets using their full address from the root (e.g. `//3rdparty/python:tool`). This is necessary to distinguish address specs from local or VCS requirements.

The lockfile will be validated against the requirements - if a lockfile doesn&#x27;t provide the requirement (at a suitable version, if the requirement specifies version constraints) Pants will error.

If unspecified, install the entire lockfile.

</Option>
